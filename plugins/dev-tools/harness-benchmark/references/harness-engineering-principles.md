# Harness Engineering 核心原则

> 基于 OpenAI Codex 团队五个月实践提炼的原则。详细原文见 SKILL.md 中的触发内容。

## 一、上下文是稀缺资源

### 原则

智能体的上下文窗口有限。给智能体一张地图，不要给一本 1000 页的说明书。

### 反模式

- 巨大的 AGENTS.md（>500 行）试图包含一切
- 信息分散在 Google Docs、Slack、人类头脑中
- 文档无索引、无交叉链接

### 正确做法

- 入口文件 ≤100-200 行，充当"地图"角色
- 知识存储在结构化的 `docs/` 目录
- 渐进式披露：小入口 → 指引 → 深层详情
- 用 linter 和 CI 验证文档质量

## 二、约束即速度

### 原则

严格的边界不会拖慢智能体，反而让它跑得更快。架构约束是智能体高效工作的前提条件。

### 反模式

- 没有依赖方向限制，代码可以随意引用
- 架构规则仅存在于人的头脑中
- 靠 code review 来维持架构一致性

### 正确做法

- 分层架构，依赖方向机械可验证
- 自定义 linter 强制执行结构规则
- 结构测试验证边界
- 约束编码后，立即作用于所有代码

## 三、可验证 > 可解释

### 原则

智能体能检查、验证的东西比它能理解的东西更有价值。将系统的更多部分转化为可机械验证的形式。

### 反模式

- 验收标准是文字描述而非可执行测试
- 质量靠人工目视检查
- Bug 修复后无法自动验证

### 正确做法

- 验收标准 = 可执行测试用例
- 应用可观测性对智能体开放（日志、指标、追踪）
- UI 可通过 Chrome DevTools 协议检查
- 修复后自动运行验证

## 四、智能体看不到的东西就不存在

### 原则

存储在 Google Docs、聊天记录或人类头脑中的知识对智能体不可见。代码仓库本地的、版本化的工件就是它所能看到的全部。

### 反模式

- 重要决策仅记录在 Slack 讨论中
- 架构模式仅存在于团队共识中
- 产品需求仅存在于项目管理工具中

### 正确做法

- 所有决策编码到仓库中的 Markdown 文件
- 设计文档有索引和验证状态
- 计划是一流工件（版本控制、集中存放）
- 产品原则文档化

## 五、熵是常数，需要垃圾回收

### 原则

完全自主的智能体会复制已有模式，包括不理想的模式。没有清理机制，代码库必然漂移。

### 反模式

- 人工每周花 20% 时间手动清理"AI 残渣"
- 技术债务不断累积直到需要大规模重构
- 不良模式在代码库中传播数周

### 正确做法

- "黄金原则"编码到代码库
- 定期运行后台任务扫描偏差
- 自动发起有针对性的重构 PR
- 像垃圾回收一样持续小额偿还技术债务

## 六、人类注意力是瓶颈

### 原则

人类的时间和注意力是固定的限制因素。所有自动化的终极目标是释放人类注意力。

### 反模式

- 人工审查每个 PR
- 测试失败需要人工排查
- 每次部署需要人工介入

### 正确做法

- 智能体对智能体的审查
- 可观测性工具对智能体开放
- 应用可通过 git worktree 独立运行
- PR 生命周期短，合并门宽松

## 七、品味编码 > 品味传达

### 原则

当文档不够完善时，将规则转化为代码。人类的品味一旦被捕捉，就持续应用于每一行代码。

### 反模式

- 编码规范仅存在于文档中
- 依赖 code review 传递品味
- 同样的反馈反复给出

### 正确做法

- 自定义 linter 规则
- 错误信息中嵌入修复指令
- 品味不变式机械执行
- 允许在边界内有表达自由

## 八、工具选择偏好"可组合"

### 原则

对智能体来说，"枯燥"的技术由于其可组合性、API 稳定性和在训练集中的表现，往往更容易建模。

### 反模式

- 选择不透明的、行为难预测的公共库
- 上游库的黑盒行为导致智能体困惑
- 过度抽象导致智能体无法推理

### 正确做法

- 倾向于可完全内化于仓库的依赖
- 必要时重新实现功能子集而非绕过不透明行为
- 优先选择 API 稳定、训练集覆盖好的技术

## 评估时的思维框架

在评估每个维度时，问自己：

1. **如果让一个新入职的智能体在这个仓库工作第一天的体验如何？**
2. **它能看到多少决策上下文？有多少是隐性的？**
3. **当它犯错时，反馈回路有多快？**
4. **当它卡住时，有什么工具可以帮助它恢复？**
5. **它产生的不理想代码有多快会被清理？**
