# 7 阶段理想 Harness 流程

> 这是 harness-benchmark 评估时使用的参考模型。评估时不是要求完全匹配，而是衡量 Skill 覆盖了哪些阶段、缺失了哪些。

## 完整流程

```
CLARIFY → SEARCH → SPEC → BLUEPRINT → TEST → IMPLEMENT → VERIFY
  ①         ②        ③        ④         ⑤        ⑥         ⑦
```

## 零干预边界（Autonomy Gate）

CLARIFY 是人类与 AI 的协作边界。CLARIFY 之后，AI 应该能完全自主运行。

```
  人类参与         │         AI 自主执行
 ──────────────── ┼ ───────────────────────────
  CLARIFY          │  SEARCH → SPEC → BLUEPRINT
  (说清楚要什么)    │  → TEST → IMPLEMENT → VERIFY
                   │
                   └──── 零干预边界（Autonomy Gate）
```

**零干预的含义**：
- CLARIFY 阶段结束后，AI 全权负责后续所有阶段
- AI 自动将长任务分解为可执行的短任务序列
- 阶段间自动推进，无需人工 approve
- 遇到错误自动诊断和回退，而非阻塞等待人类
- 人类可随时查看进度文件，但不影响执行流
- VERIFY 完成后汇报结果

## 各阶段定义

### 阶段 1: CLARIFY（明确）

```
输入：用户任务描述
输出：模块、主题、边界、目标（四要素）
门禁：四要素全部明确
关键约束：此阶段禁止写任何代码
```

**四要素**：
- **模块**：这个任务属于哪个代码模块？
- **主题**：一句话描述要做什么
- **边界**：改动范围是什么？不改什么？
- **目标**：怎样算"完成了"？

**门禁问题**：
1. 四要素是否全部有明确回答？
2. 是否理解了"为什么"要做这件事？
3. 是否理解了"不做"什么？

---

### 阶段 2: SEARCH（检索）

```
输入：四要素
输出：已有的相似实现列表 + 可复用的代码/模式
关键约束：不自由发挥，优先复用
```

**检索内容**：
1. 当前模块中是否有类似功能已实现？
2. 是否有工具函数/组件可以直接复用？
3. 是否有模式/约定需要遵循？

**为什么重要**：AI 的最大问题不是"不会写"，而是"重复造"。检索已有实现可以避免 AI 自由发挥产生不一致的代码。

---

### 阶段 3: SPEC（规格化）

```
输入：四要素 + 已有实现
输出：Spec 文件（.harness/specs/<feature>.md）
门禁：Spec 经过用户确认
```

**Spec 文件模板**：

```markdown
# Spec: {{feature-name}}

## 基本信息
- 模块：{{module}}
- 主题：{{topic}}
- 复杂度：简单/中等/复杂
- 流程级别：quick/standard/full

## 目标
{{完成定义}}

## 边界
### 在范围内
- {{item}}

### 不在范围内
- {{item}}

## 已有实现
- {{path}}: {{description}}

## 验收标准
### MUST（必须）
- [ ] {{condition}}

### SHOULD（应该）
- [ ] {{condition}}
```

---

### 阶段 4: BLUEPRINT（蓝图汇总）

```
输入：已确认的 Spec
输出：蓝图文件（.harness/blueprints/<feature>.md）
门禁：蓝图经用户确认，所有项无遗漏
```

**蓝图文件模板**：

```markdown
# Blueprint: {{feature-name}}

## 目标与边界
（从 Spec 继承，补充实现视角的边界）

## 受影响文件
| 文件 | 操作 | 说明 |
|------|------|------|
| {{path}} | 修改/新增/删除 | {{reason}} |

## 依赖关系
- {{dependency}}

## 任务清单
| # | 任务 | 依赖 | 状态 |
|---|------|------|------|
| 1 | {{task}} | - | ⏳ |
| 2 | {{task}} | 1 | ⏳ |

## 风险与回归点
| 风险 | 可能性 | 影响 | 回归策略 |
|------|--------|------|----------|
| {{risk}} | 高/中/低 | 高/中/低 | {{strategy}} |

## 失败模式
- 如果 {{condition}}：回退到 {{fallback}}
```

**核心门禁**：蓝图不齐不进入实现。这意味着：
1. 受影响文件列表不为空
2. 任务清单至少有一项
3. 风险点至少列了一条（可以是"无已知风险"）

---

### 阶段 5: TEST（测试先行）

```
输入：蓝图 + 任务清单
输出：覆盖所有 MUST 条件的测试用例
门禁：所有测试编写完成且运行失败（Red 状态）
```

**TDD 循环**：
1. **Red**：写一个失败测试，确认它真的失败
2. **Green**：写最小代码让测试通过
3. **Refactor**：在测试保护下重构

**此阶段只做 Red**：
- 为每个 MUST 条件编写测试
- 运行测试，确认全部失败
- 不写任何实现代码

---

### 阶段 6: IMPLEMENT（按清单实施）

```
输入：蓝图 + 失败测试
输出：通过测试的实现代码
关键行为：
  - 按任务清单逐项实施
  - 每完成一项 → 回写蓝图状态
  - 每完成一项 → 运行相关测试
门禁：所有测试通过（Green）
```

**实施纪律**：
1. 从蓝图任务清单的第一项开始
2. 只做让当前测试通过的最小改动
3. 完成一项 → 更新蓝图状态 → 运行测试
4. 所有测试通过后进入重构
5. 重构后测试仍然通过 → 进入 VERIFY

---

### 阶段 7: VERIFY（验证与收尾）

```
输入：全部通过的测试
输出：
  - 最终蓝图状态更新
  - 回归测试报告
  - 受影响文件清单确认
门禁：回归测试全部通过
```

**验证内容**：
1. 所有 MUST 条件的测试是否全部通过
2. 受影响文件是否与蓝图一致
3. 回归风险点是否都有覆盖
4. 最终状态回写文件

---

## 中断恢复

任何阶段中断后：
1. 所有产出已在 `.harness/` 目录（文件持久化）
2. 蓝图文件中任务状态是最新的（实时回写）
3. 新会话读取蓝图 → 从第一个 ⏳ 任务继续

## 复杂度适配

不是所有任务都需要走完 7 个阶段：

| 任务复杂度 | 推荐流程 | 跳过阶段 |
|-----------|----------|----------|
| 一句话修改（改个颜色） | quick | CLARIFY 简化 → 跳 SEARCH/SPEC/BLUEPRINT → 跳 TEST → 实现 |
| 明确的小功能 | standard | CLARIFY → SEARCH → 跳 SPEC → 简化 BLUEPRINT → TEST → IMPLEMENT → VERIFY |
| 复杂新功能 | full | 全部 7 阶段 |

**核心原则**：简单任务轻流程，复杂任务重流程。但 VERIFY 在所有模式下都不可跳过。
