# Workflow 运行模式：YOLO / Interactive（从 GSD 抽象）

本文件沉淀的是**通用运行模式设计**：不是复制 GSD 命令，而是复用其“自动化强度可切换”的封装技巧。

## 1. 核心概念

| 模式 | 定义 | 特点 |
|---|---|---|
| `yolo` | 默认自动推进，尽量减少确认点 | 效率优先，用户交互少 |
| `interactive` | 关键步骤显式确认后再推进 | 控制优先，用户交互多 |

这两种模式应该是 workflow skill 的**运行时配置**，而不是写死行为。

## 2. 何时用哪种模式

| 场景 | 推荐模式 | 原因 |
|---|---|---|
| 明确目标、低风险改动、批量执行 | `yolo` | 减少往返确认，推进快 |
| 架构选型、外部副作用、风险高 | `interactive` | 避免自动决策越界 |
| 用户首次使用该 workflow | `interactive` | 先建立信任和边界 |

## 3. 初始化时的模式选择模板

在 workflow 启动时，先问一次模式：

```markdown
请选择运行模式：
1) YOLO (Recommended for speed) — 自动批准低风险决策，快速推进
2) Interactive — 每个关键步骤先确认
```

如果用户没有明确偏好，默认 `interactive` 更稳健；如果用户明确追求速度，可切 `yolo`。

## 4. YOLO 的安全边界（必须定义）

YOLO 不是“无条件自动化”。建议至少区分：

- **可自动批准**：
  - 文档整理、重构类低风险改动
  - 已明确验收标准的重复步骤
  - 不涉及外部系统写操作的本地任务
- **必须人工确认**：
  - 生产环境变更、发布、删除/覆盖关键数据
  - 认证、权限、安全策略变更
  - 付费资源调用、第三方服务写入
  - 用户需求本身仍不明确的关键决策

## 5. 与 checkpoint / resume 的组合

即使在 `yolo`，也应保留两点：

1. `resume-signal`：关键阻塞点给明确恢复信号（如 `approved` / `next`）。
2. `Next Up`：每阶段结束后固定输出下一步命令和备选入口。

这保证了自动推进同时可控、可恢复。

## 6. 运行期切换策略

建议支持中途切换模式：

- 当用户说“先别自动做决定”时：`yolo -> interactive`
- 当用户说“后面都自动推进”时：`interactive -> yolo`

切换后应立即回显当前模式，避免误解。

## 7. 在新 skill 中的最小落地方式

1. 在 `SKILL.md` 写明两种模式定义和边界。
2. 在流程第一步加入“模式选择”。
3. 在状态文件记录 `mode` 字段，便于跨会话恢复。
4. 在高风险步骤硬编码“必须确认”规则，不受 `yolo` 覆盖。

## 8. 灵感来源

灵感来自 GSD 的 `mode: yolo|interactive` 与 checkpoint 机制；本文件已抽象为可迁移的 workflow 设计模式。
