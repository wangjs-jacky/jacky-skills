# 复杂 Skill：Shell 脚本解耦、进度外置与 Workflow 模式

适用于**步骤多、状态多、需跨会话恢复**的 skill。核心想法：**不要把全部状态与历史塞进 `SKILL.md` 正文**；用 `scripts/` 读写磁盘上的轻量事实源，模型只需「调用脚本拿摘要」。

## 1. 为何用脚本解耦

| 做法 | 问题 |
|------|------|
| 进度、分支、环境差异全写在长 Prompt 里 | 上下文膨胀、易矛盾、难维护 |
| 用脚本读写约定好的文件（JSON/纯文本） | 单一事实源；skill 正文只描述**约定与命令**，细节在盘里 |

**解耦什么**：当前阶段、已完成检查项、上次命令退出码、关联仓库路径等——凡是**可结构化**的，都适合外置。

## 2. 进度与状态：快速查询、少占上下文

- **进度文件**：例如 `.agent-state/progress.json` 或 skill 目录下 `state/current.json`（路径写在 `SKILL.md` 或 `references/` 一处，避免散落）。
- **只读查询脚本**：`scripts/status.sh`（或 `node scripts/status.mjs`）输出**短摘要**（当前 phase、未完成项数量、阻塞原因）。模型执行时先跑脚本，比让用户粘贴大段日志更省 token。
- **写入与迁移**：状态格式变更时提供一次性迁移或小段说明，避免脚本与文档长期不一致。

**节省的是什么**：主要是 **上下文中的重复叙述与手工粘贴**；把「事实」交给文件 + 脚本，对话里保留**决策与下一步**即可。

## 3. Workflow 设计模式（可组合）

以下为常见套路，可按需写在 skill 的「执行流程」里，细节参数见 `scripts/` 注释。

1. **管道（Pipeline）**  
   步骤 A → B → C，上一步产物路径固定，下一步脚本只读该路径。适合线性任务。

2. **状态机（Phase + 外置 state）**  
   `phase` 存在 JSON；每步脚本校验当前 phase，非法则退出非零并打印「应先执行 X」。适合有严格顺序的长任务。

3. **查询 / 变更分离**  
   `status` / `plan` 类脚本只读；`apply` / `advance` 类脚本才写状态。减少误触改写。

4. **幂等与重试**  
   脚本可重复执行：已完成的步骤检测到产物存在则跳过。适合网络或外部服务不稳定场景。

5. **薄编排、厚脚本**  
   `SKILL.md` 只写「何时调用哪条命令、期望什么退出码」；分支与错误文案在脚本里。避免 skill 正文变成另一份代码。

## 4. 与目录约定

- `scripts/`：可执行入口（保证 `chmod +x` 或在文档中写 `bash scripts/...`）。
- `references/`：状态文件 schema、字段含义、示例 JSON（便于人类与模型对齐）。
- 敏感信息：仍遵守 `upstream-guide.md` 中的安全扫描思路，勿把密钥写进 state 模板。

## 5. 与 XML / GSD 词汇的关系

- `gsd-xml-tags.md` 描述**文档内**阶段与 checkpoint；**外置进度**是运行时事实，二者配合：XML/Markdown 写「应有哪些 phase」，脚本与 state 文件写「当前落在哪一 phase」。
